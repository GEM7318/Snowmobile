Intro: Connector ¶

Section Assumptions

The below assumes the following about the contents of snowmobile.toml :

  1. The provided sections [connection.credentials.creds1] and [connection.credentials.creds2] are:

    1. Populated with valid credentials

    2. The first and second credentials stored respectively

    3. Aliased as creds1 and creds2 respectively

  2. The default-creds field has been left blank

1. Connecting to Snowflake ¶

Once a valid set of credentials has been stored in snowmobile.toml , a connection can be made with: 

1
2
3
4
5
6
7
 
"""
Establish connection and explore basic snowmobile.Connector attributes.
../examples/mod_connector/intro_connector.py
"""
import snowmobile

sn = snowmobile.Connect()

Several things are happening behind the scenes upon execution of line 7 in the provided snippet and any exceptions that are raised should provide direct feedback as to what's causing them.

  1. snowmobile will traverse your file system from the ground up searching for a file called snowmobile.toml . Once found, it will cache this location for future reference and not repeat this step unless the file is moved.

    • on-failure expects FileNotFoundError

  2. It will then instantiate the contents of the configuration file as pydantic objects. This ensures instant exceptions will be thrown if any required fields are omitted or unable to be coerced into their intended type.

    • on-failure expects ValidationError

  3. Once validated, it will then pass the parsed arguments to the snowflake.connector.connect() method and instantiate the SnowflakeConnection object.

    • on-failure expects DataBaseError

Since creds1 is the first set of credentials stored in snowmobile.toml and the default-creds field has been left blank (re: Section Assumptions ), line 7 is implicitly invoking: 

8
 
sn2 = snowmobile.Connect(creds='creds1')

Here's some context on how we should think about these two Connector objects: 

10
11
12
 
sn.cfg.connection.current == sn2.cfg.connection.current  #> True
sn.sql.current('schema') == sn2.sql.current('schema')    #> True
sn.sql.current('session') == sn2.sql.current('session')  #> False

Tip: Naming Convention

The following mapping of variable or attribute name to associated object is applied throughout snowmobile 's documentation and source code, including in method signatures:

  • sn : snowmobile.Connector

  • cfg : snowmobile.Configuration

  • con : snowflake.connector.SnowflakeConnection

  • cursor : snowflake.connector.cursor.SnowflakeCursor

These are some of attributes/properties on the Connector we just created:

type(sn.cursor)  #> snowflake.connector.cursor.SnowflakeCursor

2. Executing raw SQL ¶

Executing raw SQL directly off the Connector can be done in the following three ways : 

22
23
 
df1 = sn.query('select 1')  #  == pd.read_sql()
type(df1)                   #> pandas.core.frame.DataFrame

25
26
 
cur1 = sn.ex('select 1')    #  == SnowflakeConnection.cursor().execute()
type(cur1)                  #> snowflake.connector.cursor.SnowflakeCursor

28
29
 
dcur1 = sn.exd('select 1')  #  == SnowflakeConnection.cursor(DictCursor).execute()
type(dcur1)                 #> snowflake.connector.DictCursor

Alternatively, the attributes/properties of the Snowflake Connector for Python remain public so that no functionality is lost from it or supporting libraries.

Said another way, the three statements below manually implement the execution component of the three methods called on lines 22 , 25 , and 28 above, using snowmobile only as a parameter and an accessor to methods from the pandas and snowflake.connector APIs: 

32
33
34
 
import pandas as pd

df2 = pd.read_sql(sql='select 1', con=sn.con)

36
 
cur2 = sn.con.cursor().execute('select 1')

38
39
40
 
from snowflake.connector import DictCursor

dcur2 = sn.con.cursor(cursor_class=DictCursor).execute('select 1')

Which can be directly compared to our original df1 , cur1 , and dcur1 : 

42
43
44
 
print(df2.equals(df1))                       #> True
print(cur1.fetchone() == cur2.fetchone())    #> True
print(dcur1.fetchone() == dcur2.fetchone())  #> True

Final Note (sn.cursor / sn.dictcursor)

The snowmobile.Connector.cursor and snowmobile.Connector.dictcursor are properties of snowmobile.Connector that return a new instance each time they are accessed.

Depending on the desired behavior of a SnowflakeCursor or DictCursor , it it's sometimes better be store and re-referenced as opposed to instantiating new instances off the Connector object with each statement executed.

The below demonstrates the difference between executing two statements from the cursor() property compared to calling them from the same instance of cursor . 

46

cur1 = sn.cursor.execute('select 1')
cur2 = sn.cursor.execute('select 2')

cursor = sn.cursor
cur11 = cursor.execute('select 1')
cur22 = cursor.execute('select 2')

id(cur1) == id(cur2)     #> False
id(cur11) == id(cur22)   #> True
