Using nm and obj ¶

Many snowmobile.SQL methods need to know an in-warehouse object's name and type (i.e. dummy_table and table or sandbox and schema ).

These are typically passed to methods as keyword arguments, but there are times when setting these as attributes on the snowmobile.SQL object itself can keep code dry if calling multiple methods on the same in-warehouse object within another function or method.

The below is a series of small examples in which the same method is called on two instances of snowmobile.SQL , one in which these attributes are left as defaults, and the other that has had these values explicitly set on it; note that each method called on the two instances of SQL produce the same results .

The below setup code does the following:

  1. Instantiates two instances of snowmobile.Connector with the default set of credentials

  2. Sets the auto_run attribute on both instances of sql to False ; omits executing the generated commands throughout example

  3. Creates a transient sample_table table from the first, noting that this can be accessed from either session as if it were a permanent table

  4. Explicitly sets the following on the second instances of Connector :

    1. nm to sample_table

    2. obj to table

The default obj is table so this isn't necessary but included for clarity of how it should be used for other object types.

"""
Demonstrate setting attributes on the snowmobile.SQL object in place of
explicitly passing as keyword arguments.
../docs/examples/mod_sql/sql_working_example2.py
"""
import snowmobile

sn1 = snowmobile.Connect()
sn2 = snowmobile.Connect()

for sn in [sn1, sn2]:
	sn.sql.auto_run = False

sn1.ex('create transient table sample_table as select 1 as sample_col')

sn2.sql.nm = 'sample_table'
sn2.sql.obj = 'table'
